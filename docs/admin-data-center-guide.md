# 数据中台使用手册 v1

## 一句话定位

数据中台不是单纯的后台表格，而是把“用户是否在线、访问是否增长、结果是否生成、事件是否可靠落库、人格结果如何分布”变成可观察证据的工程工作台。

配套指标定义见 [`admin-metric-dictionary.md`](admin-metric-dictionary.md)。这份手册偏“怎么用”，指标字典偏“每个数字到底是什么意思、什么时候会误读”。

## 打开入口

| 项目 | 值 |
| --- | --- |
| 本地电脑端入口 | `http://127.0.0.1:5175/admin` |
| 后台 token | `dev-token`，本地验证用 |
| 默认口径 | 关闭“包含测试流量”，默认排除 `perf-test` |
| 配套字典 | [`admin-metric-dictionary.md`](admin-metric-dictionary.md) |

数据中台主要面向电脑端观察；当前版本采用左侧导航分页，避免把所有统计模块瀑布式堆在一个长页面里。移动端保留基础可读性，但不是本轮精修重点。

## 打开后先看什么

1. **实时概览**：先看当前在线用户、今日 PV、今日生成结果数和系统健康状态，判断站点现在是否有人、数据是否在进来、采集链路是否健康。
2. **流量趋势**：再看累计 PV/UV、首页访问、平均完成耗时和日趋势，判断访问和结果生成是否同步变化。
3. **人格分布**：看 120 人格分流、热门星官和五行组合，判断匿名用户群体最终落在哪些结果类型。
4. **采集链路**：看前端埋点、异步队列、MySQL 聚合、Redis ZSET 和后台 API 之间是否形成闭环。
5. **系统健康**：最后看运行态、聚合动作、短链服务和短链样本，用于排障和面试演示。

## 新增可视化怎么看

数据中台已经从“信息零碎的后台页面”重构成桌面端工程监控面板，五个页面可以这样读：

| 区域 | 看什么 | 怎么用 |
| --- | --- | --- |
| 实时概览 | 当前在线用户、今日 PV、今日生成结果、系统健康、在线趋势、异步队列摘要 | 面试或日常打开后台先看这一页，证明在线心跳、实时趋势和运行态都能闭环展示 |
| 流量趋势 | 累计 PV/UV/UIP、首页访问、平均完成耗时、完成率、日趋势、渠道与 Campaign | 判断访问增长、结果生成和完成效率，不把分享漏斗放在第一优先级 |
| 人格分布 | 120 人格分流、热门人格、星官排行、五行组合、最近生成结果 | 用 `user_result` 做匿名群体分布分析，后续可对接 Redis ZSET 实时榜单 |
| 采集链路 | 前端埋点、VisitEventService 异步队列、MySQL 聚合、Redis 缓存/ZSET、Admin API | 用工程视角讲清楚从用户行为采集到数据中台展示的链路 |
| 系统健康 | 访问事件 runtime、聚合按钮、短链服务状态、短链样本和 CSV 导出 | 用于排障、上线巡检和演示后台运维能力 |

核心使用顺序建议是：先看实时概览确认系统还活着，再看流量趋势判断访问质量，然后看人格分布观察结果分流，最后进入采集链路和系统健康做工程排障。

## 日常看数 SOP

每天或每次活动后，建议按这个顺序扫一遍：

1. **先看实时概览**：确认当前在线用户、今日 PV、今日生成结果数和系统健康状态，先判断“现在有没有人、事件有没有进来、服务是否健康”。
2. **再看流量趋势**：关注 PV/UV、首页访问、平均完成耗时和日趋势，判断访问规模和生成结果是否同向变化。
3. **进入人格分布**：观察 120 人格命中、热门星官和五行组合，判断匿名用户群体最终被分流到哪些结果类型。
4. **核对采集链路**：从前端埋点、异步队列、MySQL 聚合、Redis 缓存/ZSET 到 Admin API 顺一遍，确认数据闭环没有断点。
5. **最后看系统健康**：用 runtime、聚合按钮、短链状态、短链样本和 CSV 导出做排障或面试演示。

## 测试流量怎么处理

数据中台默认按 `visit_event.channel=perf-test` 排除压测和巡检流量。这样做的原因很简单：性能压测会集中生成结果、短链和后台查询，如果混进默认看板，完成率、回流强度和渠道排行都会被工程验证流量放大。

| 开关 | 看到什么 | 适合什么时候用 |
| --- | --- | --- |
| 关闭“包含测试流量” | 按事件 channel 排除 `perf-test` 后的运营口径 | 日常看用户是否完成、是否触发真实分享动作、短链是否有回流信号 |
| 打开“包含测试流量” | 真实用户 + 压测/巡检全量口径 | 复盘压测报告、核对压测生成的结果和短链数量 |

当前压测脚本会默认写入：

- `X-Channel: perf-test`
- `X-Campaign: performance-limit-test:<RUN_ID>`
- `User-Agent: wuxing-performance-limit-test/1.0`

如果某一天 PV、结果数或人格分布突然异常，先检查是否刚跑过压测或巡检；必要时打开“包含测试流量”做工程复核。当前隔离仍是事件查询层过滤，不是 `user_result` / `short_link` 实体层强隔离。

当前开关影响范围：

- 总览指标、趋势、渠道/Campaign 排行、人格分布、最近结果。
- 短链列表与 CSV 导出。
- 识别规则以 `visit_event.channel=perf-test` 为准；`campaign` 会带 `RUN_ID` 用于按压测批次追踪，`User-Agent` 主要用于人工排查。

## 运行态健康怎么读

后台的“访问事件运行态”是排障入口，重点不是逐个背字段，而是先看健康结论。

| 状态 | 含义 | 建议动作 |
| --- | --- | --- |
| 正常 | writer 存活、没有丢弃、没有批量写失败 | 继续观察核心运营指标 |
| 关注 | RocketMQ shadow、MQ 不可用但已回退、或出现少量丢弃 | 看队列水位、回退次数和访问峰值是否持续升高 |
| 危险 | writer 异常、批量写失败、队列接近满载、MQ 无回退 | 优先查数据库压力、队列容量、MQ 配置和服务日志 |

## 分页导航怎么读

新版后台不再把所有模块瀑布式堆在一个页面里，而是用左侧目录把数据拆成五个稳定工作台：

| 页面 | 适合回答的问题 | 面试表达重点 |
| --- | --- | --- |
| 实时概览 | 现在有没有访问、在线心跳是否在跳、服务是否健康 | 在线窗口、心跳、实时趋势、运行态摘要 |
| 流量趋势 | PV/UV 是否增长，结果生成和平均完成耗时是否稳定 | 事件明细 + 日聚合 + overview 缓存 |
| 人格分布 | 真实用户群体主要落在哪些人格、星官和组合 | `user_result` 聚合与 Redis ZSET 榜单写入 |
| 采集链路 | 从前端事件到后台看板之间有没有断点 | 前端埋点、异步写入、MySQL 索引、Redis 缓存 |
| 系统健康 | 统计链路是否可靠、短链服务是否可用 | runtime、批量写失败、队列水位、导出能力 |

左侧目录的价值是把后台从“长报表”变成“可讲清楚的工程工作台”：先看现状，再看趋势，再看分布，最后排障。

## 人格分布怎么读

人格分布页是这个项目区别于通用 PV 看板的核心页面。它不只展示访问量，而是展示用户最终被分流到哪一种人格结果里。

| 模块 | 事实来源 | 怎么读 |
| --- | --- | --- |
| 热门人格 | `user_result.persona_label` 或兜底 `persona_type_id` | 看哪些人格最容易被命中，适合后续做内容优化和分享文案复盘 |
| 热门星官 | `user_result.star_officer_name` | 看星官意象是否集中，帮助判断结果页内容是否有记忆点 |
| 五行组合 | 主副元素组合 | 看用户群体的五行倾向是否明显偏向某些结构 |
| 120 人格分流 | `persona_label/persona_type_id` 分组 | 验证“上千种答题输入最终稳定分流到 120 类人格”的核心规则 |

Redis ZSET 已经在结果生成时写入热门人格、热门星官和五行组合榜单；当前页面仍以 MySQL 聚合为权威日期口径，后续可以把 Redis 实时榜单作为单独的“实时热榜”展示。

## RocketMQ 指标怎么读

当前项目的 RocketMQ 是“可选接入点 + shadow 观察”，不是生产级完整接入。

| 字段 | 人话解释 |
| --- | --- |
| `asyncMode=local` | 默认模式，本地队列削峰后批量写库 |
| `asyncMode=rocketmq` | 会尝试发布 MQ，并根据配置回退或 shadow 本地写 |
| `rocketMqPublishedEvents` | MQ 发布成功了多少事件 |
| `rocketMqPublishFailures` | MQ 发布失败次数 |
| `rocketMqFallbackEvents` | MQ 失败后回到本地队列的次数 |
| `rocketMqShadowLocalEvents` | MQ 发布成功，但仍然写本地队列的次数 |
| `rocketMqConsumerEnabled` | 配置上是否允许 MQ consumer 接管 |
| `rocketMqConsumerPersistenceReady` | 代码实现是否确认 consumer 落库已就绪 |

安全边界：只有 `rocketMqConsumerEnabled=true` 且 `rocketMqConsumerPersistenceReady=true` 时，MQ 才允许接管落库；否则数据中台仍由本地队列保障。

## 统计口径怎么读

| 口径 | 事实来源 | 使用场景 |
| --- | --- | --- |
| `live_event` | `visit_event` 明细实时聚合 | 今天、未聚合日期、临时排查 |
| `daily_metric` | `site_daily_metric` / `short_link_daily_metric` | 已闭合历史日期，适合运营复盘 |
| `external` | 外部短链平台返回的 PV/UV/UIP | external 短链模式下的第三方统计 |
| `mixed` | 日聚合 + 实时事件混合 | 日期范围横跨已聚合历史和实时日期 |

短链历史列表现在用 `site_daily_metric` 判断日期水位：只有范围内每一天都已聚合，才读短链日聚合；否则回退实时明细。零访问短链不会再被误判成“聚合缺失”。

单条短链详情页会继续继承日期、`includeSynthetic`、关键词和统计来源上下文。访问明细本身按 `page/pageSize` 分页拉取，适合排查某条短链是否被压测、重复访问或某个渠道放大；点击“带筛选返回后台”会把原口径带回 `/admin`。

当使用 `statSource=local` 或 `statSource=external` 过滤短链时，后台会按短链创建时间分页扫描完整日期范围，再按实际统计来源过滤和分页。这个筛选不再是“最近 500 条”的近似口径；但它仍然是管理端诊断能力，不应替代离线 BI 或长期大范围报表。

后台总览有 45 秒短缓存，用来避免频繁刷新重复做聚合查询。缓存 key 带版本号；手动执行日聚合成功后会递增 overview 缓存版本，因此聚合后的下一次总览读取会直接走新口径，不需要等旧缓存自然过期。

注意：当默认排除 `perf-test` 时，看板会优先使用 `live_event` 明细实时聚合，因为历史日聚合表目前是全量口径；只有打开“包含测试流量”且日期已闭合聚合，才会回到 `daily_metric`。

## 常见异常怎么判断

| 现象 | 先看什么 | 可能原因 |
| --- | --- | --- |
| 完成率超过 `100%` | 是否刚跑过压测、是否起止日期过窄 | 分母只统计开始测试，分子包含直接提交或历史回流 |
| 回流强度非常高 | 平均每条系统短链访问次数、是否被集中打开 | 压测、内部验收、单个链接被反复访问 |
| PV 有但结果少 | 漏斗的开始测试和提交成功 | 首页吸引不足、答题页阻塞、提交接口异常 |
| 结果多但分享少 | 分享区曝光、复制链接、保存分享图、系统分享 | 结果页共鸣不足，或分享入口不明显 |

## 面试表达模板

可以这样说：

> 我做的数据中台不是只展示 PV，而是把用户主流程拆成完成、真实分享动作、系统短链回流热度和运行态四条观察线。访问事件走异步队列，降低短链跳转延迟；历史数据用日聚合减轻主库压力；运行态面板会暴露队列、丢弃、写入失败、RocketMQ shadow 和回退状态。当前 RocketMQ 是可选接入点，还没有宣称生产级接入，真正接管前必须补 consumer、eventId 幂等、死信和重聚合。

面试时不要夸大：

- 不说“强一致统计”，要说“低价值运营数据最终一致”。
- 不说“已经完整接入 RocketMQ”，要说“已经具备可插拔投递和 shadow 验证”。
- 不说“日聚合解决所有查询压力”，要说“已闭合历史日期优先聚合，未聚合日期回退实时明细”。
