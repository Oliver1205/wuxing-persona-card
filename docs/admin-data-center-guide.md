# 数据中台使用手册 v1

## 一句话定位

数据中台不是单纯的后台表格，而是把“用户是否愿意测、是否完成、是否分享、朋友是否回流”变成可观察证据的运营工作台。

配套指标定义见 [`admin-metric-dictionary.md`](admin-metric-dictionary.md)。这份手册偏“怎么用”，指标字典偏“每个数字到底是什么意思、什么时候会误读”。

## 打开入口

| 项目 | 值 |
| --- | --- |
| 本地电脑端入口 | `http://127.0.0.1:5175/admin` |
| 后台 token | `dev-token`，本地验证用 |
| 默认口径 | 关闭“包含测试流量”，默认排除 `perf-test` |
| 配套字典 | [`admin-metric-dictionary.md`](admin-metric-dictionary.md) |

数据中台主要面向电脑端观察，不按手机竖屏后台设计；如果窗口较窄，页面会保留桌面画布并横向滚动。

## 打开后先看什么

1. **测算完成**：判断首页和答题页是否顺畅，完成率过低时优先排查测试页交互和提交接口。
2. **结果分享**：判断结果页是否有分享动力，重点看分享区曝光、复制链接、保存分享图、系统分享等动作；短链生成是系统绑定能力，不等同用户主动分享。
3. **分享回流**：判断系统短链 `/s/{code}` 是否产生回流信号；是否来自朋友传播，还要结合真实分享动作、IP/设备和渠道一起判断。
4. **数据口径**：区分实时事件、日聚合和外部统计，避免把不同来源的数据直接混在一起比较。

## 新增可视化怎么看

数据中台已经从“表格后台”增强成桌面端运营面板，新增区域可以这样读：

| 区域 | 看什么 | 怎么用 |
| --- | --- | --- |
| 快捷日期与口径条 | 当前日期范围、是否排除 `perf-test`、统计来源、聚合水位和页面刷新时间 | 打开页面先确认自己看的到底是哪套口径、哪个时间点的数据，避免把压测或旧聚合误当真实运营 |
| 证据索引 | 转化、趋势、来源、短链、运行态五类证据的当前摘要 | 从顶部直接跳到对应证据区块，适合电脑端演示和排障定位 |
| 运营诊断卡 | 完成率、真实分享动作、回流强度、采样结构 | 先判断今天的问题在入口、结果页还是传播链路 |
| 趋势视图 | PV、结果生成、短链访问的日趋势和日环比 | 看增长或下跌发生在哪一天，并用 ΔPV / Δ结果 / Δ回流判断变化方向 |
| 漏斗强弱 | 首页访问、结果生成、分享动作、短链访问 | 找出用户在哪一步流失 |
| 转化链路诊断 | 相邻步骤保留率、流失数、异常倒挂提示 | 直接看入口、开始、提交、结果、分享回流哪一段最该优先处理 |
| 当前页统计来源结构 | 当前短链列表页的本地统计与外部统计占比 | 判断这一页短链主要来自站内还是第三方短链 |
| 口径差异 | 默认按 `visit_event.channel=perf-test` 过滤后的运营口径与包含测试流量的增量差 | 判断压测是否放大了 PV、结果、系统短链或回流访问 |
| 风险与行动建议 | 完成率、分享动作、回流强度、运行态和外部短链状态的综合判断 | 先按 P0/P1/P2 处理异常，再进入明细定位 |
| 复盘摘要 | 当前筛选范围下的口径、核心数据、传播、趋势、来源结构、优先动作和运行态 | 电脑端一键复制成阶段汇报或压测复盘文字，减少人工拼口径 |
| 运营雷达 | 完成力、分享意愿、回流热度、口径可信四个 0-100 观察值 | 让电脑端快速判断当前运营盘面偏弱的是体验、分享、传播还是数据可信度 |
| 五行人格分布 | 主次元素组合数量 | 看哪些人格结果最常出现 |
| 渠道与 Campaign 排行 | 不同来源的访问贡献 | 判断投放、分享渠道和活动参数效果 |
| 短链分页列表 | 当前筛选范围内的短链明细、统计口径和详情入口 | 用分页继续往下查，不再只能看第一页样本；进入详情后可继续翻访问明细 |

核心使用顺序建议是：先看运营诊断卡和风险建议，再点“定位证据”跳到漏斗、趋势、运行态或短链明细，最后进入渠道、Campaign 和短链排行定位具体来源。

布局说明：数据中台按电脑端使用设计。窄窗口或手机浏览器会保留桌面画布并横向滚动，避免把短链表格、漏斗、运行态和风险建议压成难读的竖版后台。

## 日常看数 SOP

每天或每次活动后，建议按这个顺序扫一遍：

1. **先确认口径**：先看顶部口径条，默认保持“包含测试流量”关闭，这代表按 `perf-test` 事件过滤后的运营口径；同时确认“刷新于”时间，只有复盘压测或巡检时才打开测试流量。
2. **看完成链路**：完成率低时先看首页访问、开始测试、提交成功三段漏斗，判断是入口吸引力问题还是答题页体验问题。
3. **看分享链路**：真实分享动作少，优先检查结果页是否让人愿意分享；短链访问高但回流再测少，优先检查朋友点进来后的引导。
4. **看渠道来源**：活动投放后看 Channel 和 Campaign 排行，判断哪类入口带来真实测算，而不是只看 PV。
5. **看短链明细**：用短链列表分页继续查完整筛选范围，必要时切换每页数量，再进入单条短链详情；详情页会继承日期和“包含测试流量”口径，访问明细也支持分页，返回后台时可保留原筛选。
6. **看运行态**：如果数据突然断崖或异常高，检查访问事件运行态、队列、丢弃和批量写入失败。

## 测试流量怎么处理

数据中台默认按 `visit_event.channel=perf-test` 排除压测和巡检流量。这样做的原因很简单：性能压测会集中生成结果、短链和后台查询，如果混进默认看板，完成率、回流强度和渠道排行都会被工程验证流量放大。

| 开关 | 看到什么 | 适合什么时候用 |
| --- | --- | --- |
| 关闭“包含测试流量” | 按事件 channel 排除 `perf-test` 后的运营口径 | 日常看用户是否完成、是否触发真实分享动作、短链是否有回流信号 |
| 打开“包含测试流量” | 真实用户 + 压测/巡检全量口径 | 复盘压测报告、核对压测生成的结果和短链数量 |

当前压测脚本会默认写入：

- `X-Channel: perf-test`
- `X-Campaign: performance-limit-test:<RUN_ID>`
- `User-Agent: wuxing-performance-limit-test/1.0`

如果某一天数据突然出现异常高的完成率或回流强度，先看运营诊断卡里的“测试流量排除”提示，再决定是否打开“包含测试流量”核对工程侧验证数据。

后台的“口径差异”诊断带会自动对比两套 overview：

- 默认运营口径：按事件 channel 排除 `perf-test`。
- 全量口径：包含 `perf-test`。

如果诊断带显示 `PV +300`、`结果 +20` 这类数字，代表这些增量来自测试或压测标记，不应直接用来判断真实用户增长。当前这仍是事件查询层过滤，不是 `user_result` / `short_link` 实体层强隔离。

当前开关影响范围：

- 总览指标、趋势、漏斗、渠道/Campaign 排行、人格分布、最近结果。
- 短链列表与 CSV 导出。
- 识别规则以 `visit_event.channel=perf-test` 为准；`campaign` 会带 `RUN_ID` 用于按压测批次追踪，`User-Agent` 主要用于人工排查。

## 运行态健康怎么读

后台的“访问事件运行态”是排障入口，重点不是逐个背字段，而是先看健康结论。

| 状态 | 含义 | 建议动作 |
| --- | --- | --- |
| 正常 | writer 存活、没有丢弃、没有批量写失败 | 继续观察核心运营指标 |
| 关注 | RocketMQ shadow、MQ 不可用但已回退、或出现少量丢弃 | 看队列水位、回退次数和访问峰值是否持续升高 |
| 危险 | writer 异常、批量写失败、队列接近满载、MQ 无回退 | 优先查数据库压力、队列容量、MQ 配置和服务日志 |

## 风险与行动建议怎么读

“风险与行动建议”不是新的统计口径，而是把页面上已经存在的信号合成一个处理队列。它适合在电脑端后台第一屏快速扫读：

| 优先级 | 常见触发 | 先做什么 |
| --- | --- | --- |
| `P0` | 访问事件 writer 异常、批量写失败、队列危险 | 先停下运营判断，排查写库、队列、MQ 或服务日志 |
| `P1` | 完成率超过 `100%`、外部短链不可达、运行态处于关注 | 先校准数据口径或确认降级是否生效 |
| `P2` | 分享曝光有了但真实分享动作少、回流强度异常偏高、存在测试流量差异 | 再看结果页体验、短链明细和压测标记 |
| `OK` | 已记录真实分享动作或短链回流 | 继续按渠道、Campaign 和人格组合做运营分析 |

这一区域的价值是减少“看见很多数字但不知道先处理什么”。如果建议里出现 `P0` 或 `P1`，应先处理数据可信度；如果只有 `P2` 或 `OK`，再进入产品体验和内容优化。

每条建议右侧会给出一个“定位证据”入口。这个入口不是重新计算数据，而是把当前判断带你跳到最相关的证据区块：

| 建议类型 | 默认跳转 | 目的 |
| --- | --- | --- |
| 运行态异常 | 访问事件运行态 | 先看 writer、队列、丢弃、批量写失败和 RocketMQ shadow 状态 |
| 完成率异常或完成率偏低 | 转化链路诊断 | 看开始、提交、结果和分享之间哪一步掉点 |
| 分享或回流问题 | 来源排行或短链明细 | 看动作是否来自真实分享，短链访问是否被某个链接或渠道放大 |
| 测试流量差异 | 口径差异 | 判断 `perf-test` 是否正在放大指标，避免把压测当运营增长 |

## 复盘摘要怎么用

“复盘摘要”不是新指标，而是把当前筛选范围内已经出现的关键事实整理成一段可复制文字。它适合三种场景：

| 场景 | 摘要价值 | 注意点 |
| --- | --- | --- |
| 每日运营回看 | 快速说明今天看的是哪个日期、哪套口径、核心转化和分享回流情况 | 先确认是否排除 `perf-test`，再复制给别人看 |
| 压测复盘 | 把测试批次影响、短链回流、运行态和优先动作串成一段记录 | 摘要只能辅助复盘，最终证据仍以压测报告目录为准 |
| 面试演示 | 展示数据中台不是堆表格，而是能自动提炼判断和下一步动作 | 不要把本地数据讲成线上生产数据 |

摘要里的“优先动作”来自风险与行动建议，“运行态”来自访问事件 runtime。复制后如果要外发，建议保留口径句，因为这句说明了日期范围、是否包含测试流量、统计来源、聚合水位和刷新时间。

## 运营雷达怎么读

“运营雷达”是派生观察值，不是新的事实来源。它把页面已有指标压成四个 0-100 分值，用来给电脑端后台一个更快的扫盘入口：

| 维度 | 来自哪里 | 怎么读 |
| --- | --- | --- |
| 完成力 | `completionRate` | 结果生成相对开始测试的强度；超过 `100%` 时会降为关注态，因为应先校准口径 |
| 分享意愿 | 复制链接、保存分享图、系统分享动作 / 结果生成 | 分数低代表用户看到结果后还不够愿意主动传播 |
| 回流热度 | `shortLinkVisits / shortLinkCreated` | 观察每条系统短链平均带来的访问；过高时先排除压测或重复访问 |
| 口径可信 | `metricSource`、测试流量差异、是否排除 `perf-test` | 分数不是“数据绝对正确”，而是提示当前口径是否适合拿去汇报 |

雷达分数用于快速发现方向，不用于替代下方明细。真正排障时仍要回到漏斗、趋势、渠道、Campaign、短链列表和运行态。

## 转化链路诊断怎么读

“转化链路诊断”读取的是同一组 `funnelSteps`，不是后端新增统计口径。它把漏斗从“每一步有多少次”翻译成“相邻两步保留了多少、流失了多少”。

| 看法 | 含义 | 优先动作 |
| --- | --- | --- |
| 保留率高 | 上一步到这一步基本顺畅 | 继续看下一段，不急着改这一段 |
| 保留率中等 | 有可见流失，但还不是断崖 | 结合页面截图、按钮位置、提交日志继续判断 |
| 保留率低 | 当前段是明显体验断点 | 优先复查页面交互、接口错误、移动端布局和文案吸引力 |
| 后一步高于前一步 | 数据倒挂 | 先排查压测、接口直调、日期范围过窄或事件口径不完全同源 |

这一区块适合电脑端快速扫盘：如果“开始测试 -> 提交尝试”掉得最多，优先看答题页；如果“结果生成 -> 分享动作”掉得最多，优先看结果页的分享动机和按钮。

## RocketMQ 指标怎么读

当前项目的 RocketMQ 是“可选接入点 + shadow 观察”，不是生产级完整接入。

| 字段 | 人话解释 |
| --- | --- |
| `asyncMode=local` | 默认模式，本地队列削峰后批量写库 |
| `asyncMode=rocketmq` | 会尝试发布 MQ，并根据配置回退或 shadow 本地写 |
| `rocketMqPublishedEvents` | MQ 发布成功了多少事件 |
| `rocketMqPublishFailures` | MQ 发布失败次数 |
| `rocketMqFallbackEvents` | MQ 失败后回到本地队列的次数 |
| `rocketMqShadowLocalEvents` | MQ 发布成功，但仍然写本地队列的次数 |
| `rocketMqConsumerEnabled` | 配置上是否允许 MQ consumer 接管 |
| `rocketMqConsumerPersistenceReady` | 代码实现是否确认 consumer 落库已就绪 |

安全边界：只有 `rocketMqConsumerEnabled=true` 且 `rocketMqConsumerPersistenceReady=true` 时，MQ 才允许接管落库；否则数据中台仍由本地队列保障。

## 统计口径怎么读

| 口径 | 事实来源 | 使用场景 |
| --- | --- | --- |
| `live_event` | `visit_event` 明细实时聚合 | 今天、未聚合日期、临时排查 |
| `daily_metric` | `site_daily_metric` / `short_link_daily_metric` | 已闭合历史日期，适合运营复盘 |
| `external` | 外部短链平台返回的 PV/UV/UIP | external 短链模式下的第三方统计 |
| `mixed` | 日聚合 + 实时事件混合 | 日期范围横跨已聚合历史和实时日期 |

短链历史列表现在用 `site_daily_metric` 判断日期水位：只有范围内每一天都已聚合，才读短链日聚合；否则回退实时明细。零访问短链不会再被误判成“聚合缺失”。

单条短链详情页会继续继承日期、`includeSynthetic`、关键词和统计来源上下文。访问明细本身按 `page/pageSize` 分页拉取，适合排查某条短链是否被压测、重复访问或某个渠道放大；点击“带筛选返回后台”会把原口径带回 `/admin`。

当使用 `statSource=local` 或 `statSource=external` 过滤短链时，后台会按短链创建时间分页扫描完整日期范围，再按实际统计来源过滤和分页。这个筛选不再是“最近 500 条”的近似口径；但它仍然是管理端诊断能力，不应替代离线 BI 或长期大范围报表。

后台总览有 45 秒短缓存，用来避免频繁刷新重复做聚合查询。缓存 key 带版本号；手动执行日聚合成功后会递增 overview 缓存版本，因此聚合后的下一次总览读取会直接走新口径，不需要等旧缓存自然过期。

注意：当默认排除 `perf-test` 时，看板会优先使用 `live_event` 明细实时聚合，因为历史日聚合表目前是全量口径；只有打开“包含测试流量”且日期已闭合聚合，才会回到 `daily_metric`。

## 常见异常怎么判断

| 现象 | 先看什么 | 可能原因 |
| --- | --- | --- |
| 完成率超过 `100%` | 是否刚跑过压测、是否起止日期过窄 | 分母只统计开始测试，分子包含直接提交或历史回流 |
| 回流强度非常高 | 平均每条系统短链访问次数、是否被集中打开 | 压测、内部验收、单个链接被反复访问 |
| PV 有但结果少 | 漏斗的开始测试和提交成功 | 首页吸引不足、答题页阻塞、提交接口异常 |
| 结果多但分享少 | 分享区曝光、复制链接、保存分享图、系统分享 | 结果页共鸣不足，或分享入口不明显 |

## 面试表达模板

可以这样说：

> 我做的数据中台不是只展示 PV，而是把用户主流程拆成完成、真实分享动作、系统短链回流热度和运行态四条观察线。访问事件走异步队列，降低短链跳转延迟；历史数据用日聚合减轻主库压力；运行态面板会暴露队列、丢弃、写入失败、RocketMQ shadow 和回退状态。当前 RocketMQ 是可选接入点，还没有宣称生产级接入，真正接管前必须补 consumer、eventId 幂等、死信和重聚合。

面试时不要夸大：

- 不说“强一致统计”，要说“低价值运营数据最终一致”。
- 不说“已经完整接入 RocketMQ”，要说“已经具备可插拔投递和 shadow 验证”。
- 不说“日聚合解决所有查询压力”，要说“已闭合历史日期优先聚合，未聚合日期回退实时明细”。
