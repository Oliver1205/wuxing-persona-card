# v0.2 短链接适配层设计

## 1. 版本目标

v0.2 不重做 v0.1 已经跑通的短链闭环，而是在保持现有业务可用的前提下，把短链模块从单一内置实现升级为可切换适配层。

核心目标：

- 默认模式继续使用内置短链，保证本项目独立可运行。
- 预留外部短链服务模式，后续可接入 `/Users/linyuxiang/JavaBackend/01_Projects/shortlink`。
- 五行项目继续保留结果和短链的业务绑定，避免外部服务不可用时丢失业务上下文。
- 不引入朋友匹配、登录注册、付费、AI 深度解读或复杂 BI。

## 2. 当前内置短链职责

v0.1 的 `ShortLinkService` 同时承担了以下职责：

- 生成 6 位 Base62 短码。
- 检查短码是否已存在，冲突时重试。
- 保存 `short_link` 映射：`shortCode -> resultId -> shortUrl`。
- 写入 Redis 短链解析缓存。
- 对不存在短码写入空值缓存。
- 解析 `/s/{shortCode}` 并跳回 `/result/{resultId}?sc={shortCode}`。
- 记录 `SHORT_LINK_VISIT` 事件。
- 按事件表回算并更新 PV、UV、UIP。

这些能力仍然是 MVP 的默认实现。

## 3. 外部短链服务未来职责

外部短链服务建议只承担短链基础设施能力：

- 创建短链。
- 管理短码唯一性和跳转映射。
- Redis 解析缓存。
- 无效短链空值缓存。
- 独立短链访问统计和访问日志。
- 后续提供统计查询接口。

五行项目不直接接入它的 SaaS 控制台、用户注册、网关、回收站和控制台前端。

## 4. 五行项目保留的数据

即使切到外部短链服务，五行项目仍保留 `short_link` 表作为业务绑定表：

| 字段 | 作用 |
| --- | --- |
| `short_code` | 从外部 `fullShortUrl` 解析出的短码，或内置生成的短码 |
| `result_id` | 五行测算结果 ID |
| `original_path` | 五行结果页路径 `/result/{resultId}` |
| `short_url` | 返回给前端复制分享的完整短链 |
| `pv_count` / `uv_count` / `uip_count` | 五行项目本地统计，后续可替换为外部统计聚合 |
| `last_visit_at` | 最近一次通过五行入口访问短链的时间 |

这样做的好处：

- 结果页查询不依赖外部短链服务。
- `/admin` 可以继续展示业务维度短链列表。
- 外部服务不可用时可降级到内置模式。

## 5. 分层设计

```text
ResultService
  -> ShortLinkService                  门面，保持 v0.1 调用入口不变
    -> ShortLinkProvider               统一接口
      -> InternalShortLinkProvider     默认内置实现
      -> ExternalShortLinkProvider     外部服务适配实现
        -> ExternalShortLinkClient     HTTP Client 边界
```

### ShortLinkProvider

统一定义短链能力：

```text
createForResult(resultId)
resolveAndRecord(shortCode, clientId, request)
getByResultId(resultId)
getByShortCode(shortCode)
```

### InternalShortLinkProvider

从 v0.1 的 `ShortLinkService` 中拆出，继续负责：

- 本地短码生成。
- MySQL 和 Redis 映射。
- 空值缓存。
- 本地访问事件和 PV/UV/UIP。

### ExternalShortLinkProvider

v0.2 只做可运行的外部服务预留，不强制要求外部服务启动：

- 读取配置构造外部创建短链请求。
- 调用 `ExternalShortLinkClient`。
- 从 `fullShortUrl` 解析 `shortCode`。
- 将 `resultId -> shortCode -> shortUrl` 保存到本地 `short_link` 表。
- 外部调用失败时，按配置降级到 `InternalShortLinkProvider`。
- `/s/{shortCode}` 解析和后台统计暂时继续使用本地绑定与本地事件表。

后续真实接入时，可以把 Nginx 的 `/s/` 或短链子域名切到外部短链服务；五行后台再按需读取外部统计接口。

## 6. 配置切换

默认配置：

```yaml
app:
  short-link:
    mode: internal
```

外部服务预留配置：

```yaml
app:
  short-link:
    mode: external
    external:
      base-url: http://localhost:8003
      group-id: wuxing_persona
      fallback-to-internal: true
      system-username: wuxing_system
      system-user-id: wuxing-system
      system-real-name: wuxing-system
```

说明：

- `internal`：完全使用五行项目内置短链。
- `external`：创建短链时优先调用外部服务。
- `fallback-to-internal=true`：外部服务不可用时自动使用内置短链，保证主流程不挂。
- 若后续需要强依赖外部服务，可将 `fallback-to-internal=false`，外部服务失败时返回明确业务错误。

## 7. 外部短链请求结构

创建短链请求与外部项目保持兼容：

```json
{
  "originUrl": "https://your-domain.com/result/R202606090001",
  "gid": "wuxing_persona",
  "createdType": 0,
  "validDateType": 0,
  "validDate": null,
  "describe": "五行人格卡结果 R202606090001"
}
```

响应重点字段：

```json
{
  "data": {
    "gid": "wuxing_persona",
    "originUrl": "https://your-domain.com/result/R202606090001",
    "fullShortUrl": "http://localhost:8003/abc123"
  }
}
```

## 8. v0.2 验收标准

- 默认 `internal` 模式下，v0.1 所有接口和页面行为不回退。
- `external` 模式下，外部创建成功时本地保存绑定。
- `external` 模式下，外部请求失败且允许降级时，主流程自动回到内置短链。
- `external` 模式下，外部请求失败且禁止降级时，返回明确业务错误。
- `mvn test`、前端构建和 Compose 配置校验通过。
