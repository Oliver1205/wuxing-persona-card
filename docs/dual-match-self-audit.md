# 双人五行匹配自审与设计记录

日期：2026-06-12

## 目标

让用户复制一段纯短码后，再进入首页时由前端尝试检测剪贴板内容；如果内容是可识别且存在的短码，则询问用户是否进入双人匹配模式。用户完成同一套测评后，结果直接进入双人匹配结果页，不再展示单人结果页。

## 自审结论

| 视角 | 当前隐患 | 决策 |
| --- | --- | --- |
| 产品流程 | 结果页之前更强调复制分享链接，用户不容易知道“短码”也有用途 | 分享组件增加短码复制入口，并说明短码用于双人匹配 |
| 前端交互 | 浏览器剪贴板读取可能被权限策略拒绝 | 首页先尝试非阻塞自动检测；失败不打断主流程，用户也可以点击检测或手动输入短码 |
| 输入边界 | 用户可能复制整条链接而不是短码 | 只接受 `6-7` 位 Base62 纯短码，不从 URL 中提取短码，避免误触发 |
| 后端契约 | 前端直接拼结果会导致刷新丢失或数据不一致 | 后端新增匹配查询/创建接口，匹配页用两个短码可重复加载 |
| 数据模型 | 为匹配新增表会扩大迁移和运营成本 | 第一版不持久化匹配记录，使用已有 `user_result` 和 `short_link` 即时计算 |
| 安全隐私 | 短码本身是公开分享凭据，匹配页可能被转发 | 只展示已公开的人格卡摘要，不暴露生日、答案原文或后台统计 |
| 高峰流量 | 双人匹配会多读两份结果 | 复用短码索引和结果读取链路；第一版不引入额外写放大 |

## 后端接口形态

- `GET /api/matches/candidates/{shortCode}`：验证剪贴板短码是否存在，并返回用于首页确认的轻量摘要。
- `POST /api/matches`：使用 `partnerShortCode + CreateResultRequest` 创建当前用户结果，并返回匹配结果。
- `GET /api/matches/{partnerShortCode}/{currentShortCode}`：用两个短码读取并计算匹配结果，保证匹配结果页刷新后仍可访问。

## 匹配算法原则

- 不输出决定论、运势或负面断言。
- 以五行分布相似度为主，以主副五行关系作为轻量调节。
- 文案强调“节奏”“互补”“沟通建议”，避免给关系下结论。
