# v2.0 商业级产品化方案

## 1. 阶段目标

v2.0 的目标不是继续堆功能，而是把「五行人格卡」从工程闭环 MVP 推进到可验证增长的产品闭环：

```text
用户愿意开始测试
  -> 结果让用户觉得“像我”
  -> 用户愿意分享
  -> 朋友点击短链
  -> 朋友继续生成自己的结果
```

当前短链接、匿名统计、Redis 缓存、Docker Compose、后台中台和 external Provider 仍然是项目的工程地基。v2.0 在不破坏这些基础能力的前提下，优先升级用户体验、分享动机、增长埋点和后续架构路线。

## 2. 多角色协作结论

| 角色 | 核心判断 | 本轮落地 |
| --- | --- | --- |
| 产品经理 | 最大缺口不是功能数量，而是用户分享动机和结果身份感 | 首页、测试页、结果页围绕传播闭环重做 |
| 前端体验负责人 | 测评产品要减少表单感，增强节奏、进度和结果可截图感 | 增加进度条、sticky CTA、人格身份标题、五行分布和分享入口 |
| 后端架构师 | v2.x 要把 assessment、shortlink、analytics、ops 边界逐步拆清 | 记录架构路线，并补非法出生时段业务异常 |
| 增长分析 | 只看 PV 不够，需要跟踪开始、答题、提交、分享、回流、二次测试 | 新增关键增长事件枚举和前端上报点 |
| 质量与 SRE | 商业化前必须补 E2E、备份恢复、HTTPS、限流、告警和回滚 | 写入后续路线，不在本轮冒进实现 |

## 3. 产品原则

可传播的人格测评产品通常有几个共同点：

- 首屏低门槛：告诉用户要花多久、答几题、能得到什么。
- 答题节奏清晰：有进度反馈、明确选中态和低干扰表单。
- 结果可记忆：用户能用一句话说出自己的“身份标签”。
- 解释有层次：先给身份，再给关键词，再给完整解释。
- 分享动作顺手：复制链接、系统分享、保存分享图都应该可用。
- 数据能闭环：不仅看访问量，还要看从分享到二次测试的路径。

五行人格卡可以吸收这些产品机制，但不能复制 MBTI 或其他品牌的名称、视觉资产和受保护表达，也不能做宿命论、疾病、灾祸、财富、婚姻等确定性判断。

## 4. v2.0 范围

本轮必须完成：

- 首页改为商业化首屏，突出“约 90 秒、5 道题、结果可分享”。
- 测试页增加进度、完成度、隐私说明和移动端 sticky CTA。
- 答题选项隐藏五行元素标签，减少用户按想要的结果倒推选择。
- 结果页强化人格身份、关键词、五行分布和分享入口。
- 分享面板补充原生分享能力，并保留复制短链。
- 增长事件补充到开始答题、选择答案、提交尝试、分享面板、原生分享、保存分享图、重新测试和二次测试。
- 后端将非法出生时段收敛为明确业务异常，并用测试锁住。

本轮暂不做：

- 登录、注册、用户账户。
- 付费报告。
- AI 深度解读。
- 朋友匹配或关系评分。
- 社区、评论、点赞、关注。
- 复杂 BI 大屏。
- 宿命论、运势、疾病、灾祸、财富、婚姻等预测内容。

## 5. 增长漏斗

推荐的 v2.x 漏斗口径：

```text
PAGE_VIEW_HOME
  -> START_TEST_CLICK
  -> TEST_FORM_START
  -> QUESTION_ANSWER_SELECT
  -> TEST_SUBMIT_ATTEMPT
  -> TEST_SUBMIT
  -> RESULT_CREATED
  -> RESULT_VIEW
  -> SHARE_PANEL_VIEW
  -> SHORT_LINK_COPY / SAVE_SHARE_IMAGE_SUCCESS / NATIVE_SHARE_SUCCESS
  -> SHORT_LINK_VISIT
  -> SHARED_RESULT_CTA_CLICK
  -> RESULT_CREATED
```

核心指标：

- 开始率 = `START_TEST_CLICK / PAGE_VIEW_HOME`
- 提交尝试率 = `TEST_SUBMIT_ATTEMPT / TEST_FORM_START`
- 结果生成率 = `RESULT_CREATED / TEST_SUBMIT_ATTEMPT`
- 分享参与率 = 分享事件 / `RESULT_VIEW`
- 短链回流率 = `SHORT_LINK_VISIT / 分享事件`
- 分享结果转化率 = 分享结果页二次测试 / `SHORT_LINK_VISIT`

注意：`TEST_SUBMIT` 仍由后端在结果创建流程中记录，前端只记录 `TEST_SUBMIT_ATTEMPT`，避免提交次数重复计数。

## 6. 前端体验基线

### 首页

首页需要在第一屏回答：

- 这是什么？
- 要花多久？
- 我要答多少题？
- 结果能不能分享？
- 这是不是严肃预测？

本轮首页新增测试耗时、题量、结果可分享承诺和人格卡样例，让用户更快理解价值。

### 测试页

测试页需要像轻量测评，而不是后台表单。本轮新增：

- 出生年月完成度。
- 5 道题答题进度。
- 隐私说明。
- sticky 提交区。
- A/B/C/D/E 选项标识，隐藏元素标签。

### 结果页

结果页需要先给用户一句可传播的身份：

```text
金水 · 白露星官
清醒 · 判断 · 洞察
```

然后再提供支撑信息：

- 人格卡片。
- 正向解释。
- 完整五行分布。
- 专属短链接。
- 系统分享。
- 分享图。
- 重新测试和“我也要测”入口。

## 7. 后端架构路线

当前优势：

- MVP 业务闭环完整。
- 内置 / external 短链 Provider 已抽象。
- Redis 结果缓存和短码缓存已落地。
- clientId、IP、User-Agent hash 入库。
- 后台具备总览、短链列表、访问明细、导出和 external 运行态状态。
- 已有 H2、MockMvc 和 Testcontainers 能力。

商业化风险：

- 短链访问和后台指标仍偏向从原始事件计算，数据量变大后会变慢。
- 后台短链列表存在逐行补结果和统计的成本。
- 数据库 schema 仍依赖 SQL init，缺少正式迁移工具。
- 短链表缺少 provider 元数据、有效期和 active 唯一约束。
- 事件表缺少 session、channel、campaign、device、event_date 等分析字段。
- 生产运维还缺备份恢复、回滚、HTTPS、限流、告警和正式 E2E。

推荐的 v2.x 模块边界：

- `assessment`：题目、计分、星官、文案、结果创建。
- `shortlink`：短链绑定、Provider、跳转、provider 元数据。
- `analytics`：事件采集、日聚合、漏斗指标、后台查询模型。
- `ops`：健康检查、运行态、导出、备份、部署脚本。

推荐的数据演进：

- `user_result` 增加 `rule_version`、`question_set_version`、`text_template_version`。
- `short_link` 增加 `provider_mode`、`provider_code`、`domain`、`external_link_id`、`active`、`expire_at`。
- 新增 `site_daily_metric` 和 `short_link_daily_metric`。
- `visit_event` 增加 `session_id`、`channel`、`campaign`、`device_type`、`event_date`。
- 引入 Flyway 或 Liquibase，再继续做 schema 演进。

## 8. 运维路线

商业化访问前建议补齐：

- HTTPS、HSTS 和更完整的 CSP。
- `/api/events`、`/api/results` 和无效 `/s/**` 的 Nginx / 后端限流。
- MySQL 备份、恢复脚本和恢复演练。
- Compose 部署回滚脚本。
- 真实域名 public smoke test。
- backend / nginx 容器 healthcheck 和 restart policy。
- 磁盘、内存、容器状态、健康接口的基础告警。
- CI 中的 secret scan、依赖漏洞扫描和容器镜像扫描。
