# v1.0 路线图与质量门禁

本文档用于把「五行人格卡」从 v0.5 推进到 v1.0 稳定版。v1.0 不是盲目增加功能，而是在不破坏单人测算闭环和短链业务价值的前提下，把部署、质量、稳定性、隐私和面试表达收束到可交付状态。

## 1. v1.0 总目标

v1.0 稳定版必须满足：

1. 单人测算闭环稳定：`/` -> `/test` -> `/result/:resultId` -> 短链分享 -> `/admin` 统计。
2. 短链能力稳定：internal 默认可用，external 可配置接入，创建、解析、统计和访问明细均有失败回退。
3. 部署路径清晰：Docker Compose、Nginx、环境变量、域名路由、HTTPS 前置方案和上线检查表完整。
4. 质量门禁可重复：每个版本必须能执行同一套检查脚本并记录结果。
5. 隐私边界明确：不保存明文 IP / clientId，外部访问记录返回前做 hash，不做登录注册。
6. 文案边界稳定：保持正向、娱乐、非预测，不引入宿命论或负面断言。
7. 文档可面试表达：README、教学手册、短链集成方案、质量评分和版本进度完整。

## 3. 严格质量门禁

每个版本合并前必须执行：

```bash
scripts/quality-check.sh
```

该脚本至少覆盖：

- `git diff --check`
- 构建产物未被 Git 跟踪
- 用户可见源码禁止负面宿命文案
- `cd backend && mvn -q test`
- `cd frontend && npm run build`
- `docker compose --env-file deploy/.env.example -f deploy/docker-compose.yml config`

如需做完整容器验收，再执行：

```bash
APP_BASE_URL=http://localhost:8088 \
NGINX_HTTP_PORT=8088 \
docker compose --env-file deploy/.env.example -f deploy/docker-compose.yml up --build -d
```

如果 Docker Hub 或镜像源不可用，可以记录为环境阻塞，但不能把“未执行”写成“已通过”。

## 4. 代码规范要求

后端：

- Controller 不写业务规则。
- Service 保持单一职责，跨系统调用放到适配器或 Client。
- 所有外部输入必须校验。
- Redis、external 短链服务和统计接口都必须有失败回退或明确业务错误。
- 不保存明文 IP、明文 clientId 或外部访问记录明文 user。
- 新增分支必须配套单元测试或集成测试；无法测试时必须写明原因。

前端：

- API 调用统一经过 `src/api/request.ts`。
- 业务类型写在 `src/api/types.ts`，避免页面里手写隐式结构。
- 管理后台保持轻量、可扫描，不做复杂 BI 大屏。
- 移动端主流程优先，不引入营销落地页。
- 页面展示的统计来源、错误态和空状态必须可理解。

文档：

- README 保持仓库主页级专业表达。
- 每个版本必须在“开发进度记录”增加日期条目。
- `docs/quality-scorecard.md` 必须同步自评分和未验证项。
- 设计变化必须写入对应专项文档或 `docs/project-plan.md`。

## 5. 禁止越界

v1.0 前仍不做：

- 朋友匹配
- 登录注册
- 付费系统
- AI 深度解读
- 社区、评论、点赞、关注
- 复杂八字排盘
- 复杂 BI 大屏

这些功能可以作为 v2.0 以后候选，但不能影响 v1.0 的稳定收口。

## 6. 发布流程

每个版本按以下顺序执行：

1. 从 `main` 新建 `feature/vX.Y-...` 分支。
2. 实现小范围、高内聚改动。
3. 执行 `scripts/quality-check.sh`。
4. 更新 README、版本文档、质量评分和项目计划。
5. 提交并推送 feature 分支。
6. 创建 PR，确认 diff 和验证记录。
7. 合并到 `main`。
8. 打版本标签并推送。

v1.0 标签建议：

```text
v1.0.0-stable
```
