# 真实用户与设备验收清单

本文档用于把“网页交互更自然、更符合直觉”变成可验证的观察任务。它不是自动化测试脚本，而是给产品、前端、美术和普通访问者审视使用的轻量验收表。

## 1. 验收目标

| 目标 | 观察方式 | 通过标准 |
| --- | --- | --- |
| 首屏愿意开始 | 让用户从首页进入，不解释项目背景 | 10 秒内能说出这是一个人格测试，并找到开始入口 |
| 样例后能继续 | 用户点击“先看样例”后停在样例卡附近 | 不需要回到首屏，就能看到继续生成自己卡片的入口 |
| 出生信息不劝退 | 观察用户停留和提问 | 用户理解只需要出生年月，日期和时段是可选 |
| 答题节奏可控 | 观察选中答案后的反应 | 用户知道可以手动下一题，也不会被自动前进吓到 |
| 结果有共鸣 | 让用户读结果页前两屏 | 用户能指出至少 1 句“像我”的描述 |
| 分享动作清楚 | 不提示用户，观察结果页操作 | 用户能找到保存分享图或复制分享链接入口 |
| 朋友回流自然 | 用分享链接打开同一张结果 | 用户知道这是朋友的卡，也能找到“我也测一张” |

## 2. 设备矩阵

| 设备 | 浏览器 | 必测页面 | 重点风险 |
| --- | --- | --- | --- |
| iPhone | Safari | 首页、测试页、结果页 | 底部 sticky 操作条是否遮挡最后一项 |
| iPhone | 微信内置浏览器 | 结果页、分享回流页 | 复制分享链接、保存图、返回手势 |
| 安卓中端机 | Chrome | 测试页、结果页 | 字体换行、按钮高度、滚动性能 |
| 安卓微信 | 微信内置浏览器 | 分享回流页 | 分享入口、剪贴板限制、顶部安全区 |
| 桌面浏览器 | Chrome / Edge | 文档站、后台页 | 文档站首屏、后台表格和空状态 |

## 3. 首次用户访谈脚本

每位用户只做 8 到 10 分钟，不提前解释功能。

1. 打开首页后，问：“你觉得这个页面是让你做什么？”
2. 让用户点击“先看样例”，观察是否能从样例卡继续开始测试。
3. 请用户自己开始测试，记录是否卡在出生信息。
4. 观察用户选择第一题答案后是否想修改。
5. 结果页出现后，问：“哪一句最像你？哪一句最不像？”
6. 问：“如果要发给朋友，你会点哪里？”
7. 用分享链接打开同一结果，问：“你知道这是朋友分享来的页面吗？”

记录格式：

```text
用户代号：
设备/浏览器：
首页是否 10 秒内理解：
样例后是否找到继续入口：
出生信息卡点：
答题节奏卡点：
最有共鸣的一句：
最想改的一句：
是否找到分享入口：
是否理解朋友回流：
备注：
```

## 4. 截图归档要求

每次正式验收至少保留这些截图：

- 首页首屏。
- 出生年月步骤。
- 第 1 题答题卡。
- 选中答案后的确认态。
- 结果页首屏。
- 结果页“为什么像你”三卡片。
- 结果页分享快捷操作。
- 分享回流 banner。
- 后台总览空状态和有数据状态。

截图建议输出到 `docs/screenshots/showcase/`，命名格式：

```text
YYYYMMDD-device-browser-page-state.png
```

## 5. 不通过时的处理规则

| 现象 | 优先处理 |
| --- | --- |
| 用户不知道哪里开始 | 首页首屏文案和主按钮 |
| 用户以为日期/时段必填 | 测试页出生信息说明和折叠态 |
| 用户觉得自动跳太快 | 答题确认反馈和自动前进节奏 |
| 用户觉得结果不准或太空 | 结果页共鸣句和生活场景 |
| 用户找不到分享入口 | 结果卡附近的保存图和分享链接入口 |
| 微信内置浏览器复制失败 | 手动复制兜底和保存图提示 |

## 6. 面试表达

面试时不要说“我凭感觉把交互改自然了”，可以这样说：

> 我把体验优化拆成可观察任务：用户是否理解首屏、是否被出生信息劝退、是否能控制答题节奏、结果页是否有共鸣、能否找到分享入口、朋友回流是否自然。后续每次体验迭代都会留下设备、浏览器、截图和访谈记录。
