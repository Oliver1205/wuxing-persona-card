# 前端视觉系统

记录日期：2026-06-22
适用范围：五行人格卡 H5、结果页、匹配页、分享卡、后台数据中台和 showcase 截图。

## 目标

五行人格卡的前端视觉不是“玄学装饰”，而是一个短链传播型人格测试产品。视觉系统要帮助用户快速开始、顺利答完、理解结果、愿意分享，并让后台运营界面可读、可扫、可验证。

核心方向：克制中式数据感。

这句话拆开是：

- 克制：不堆装饰、不滥用纹样、不做大面积神秘氛围。
- 中式：使用温润纸感、书写感标题、五行元素，但不走厚重古风。
- 数据感：五行比例、短链指标、访问明细和后台报表都要可扫描、可比较。

## 设计原则

1. 移动端优先：H5 用户主流程先服务手机端，桌面端自然居中。
2. 一屏一任务：引导、出生信息、题卡、结果、分享和匹配都应有清晰主行动。
3. 五行色只表达信息：五行颜色用于元素编码、比例和状态，不作为全屏装饰。
4. 后台像工具：后台以密度、筛选、对比、导出和故障定位为主，不做营销页。
5. 截图可验收：每个可见改造都要能在 iPhone SE、安卓宽屏、桌面截图里说明价值。

使用规则：

- 结果页可以使用主副元素色表达身份。
- 后台图表和筛选 chip 可以使用五行色，但必须保留文字标签，不能只靠颜色表达。
- 不使用大面积五行渐变当背景。
- 不使用红金满屏、紫蓝渐变、发光光斑或漂浮装饰球。

## 字体层级

当前字体栈：

```text
HarmonyOS Sans SC -> MiSans -> PingFang SC -> Inter -> Noto Sans SC -> Hiragino Sans GB -> Microsoft YaHei -> sans-serif
```

结果身份标题可使用：

```text
Songti SC -> STSong -> Noto Serif SC -> display fallback
```

使用规则：

- H5 首屏标题可以大，但不要像落地页营销 hero。
- 结果页身份标题强调“被看见”，但解释正文优先可读。
- 后台标题和卡片标题要小而准，避免 hero 级字号出现在工具面板里。
- 不用负字距；不使用按视口宽度缩放字体。

## 页面规则

### 首页 `GuidePage`

目标：让用户快速理解这是娱乐型人格卡，并能进入测试或粘贴短码匹配。

规则：

- 第一屏给出产品身份和主行动，不堆功能说明。
- 剪贴板、短码、手动匹配都要保持低压迫感。
- 不自动读取剪贴板。
- 所有输入和按钮保持 44px 触控目标。

### 测试页 `TestPage` / `QuestionCard`

目标：让用户像刷卡片一样完成出生信息和 5 道题。

规则：

- 一屏处理一个任务。
- 顶部进度清楚，但不抢题目注意力。
- 选项触控面积足够，选中态明显。
- 禁用按钮不能看起来像可点击主行动。
- 出生年月日控件不使用破坏中文界面的原生空值占位。

### 结果页 `ResultPage` / `PersonaCard`

目标：用户一眼知道自己是什么人格、为什么、下一步做什么。

规则：

- 身份卡优先，其次五行比例，再解释和分享。
- 五行字章是信息锚点，不画简笔图、不做神秘符号。
- 结果文案保持正向、娱乐、友好，不出现宿命论和恐吓预测。
- 普通结果页保留单一分享模块；分享回流页不出现二次分享盒。

### 分享区 `ShareLinkBox` / `shareCard.ts`

目标：分享像产品能力，不像调试工具。

规则：

- 主动作是保存分享图和系统分享。
- 复制短码、复制链接属于备用能力，可以折叠。
- 外露链接尽量干净，复制动作可保留真实归因 URL。
- 分享图保持 `900x1200` 输出稳定。

### 匹配页 `MatchPage`

目标：朋友关系解释优先于装饰图例。

规则：

- 匹配分数、双方摘要、关系参照、优势和建议有清晰顺序。
- 五行信息必须服务关系解释。
- 短码继续输入入口可用，但不能压过结果内容。

### 后台 `AdminDashboard` / `AdminShortLinkDetail`

目标：高密度、可扫描、可定位问题。

规则：

- 桌面端展示完整报表和宽表格。
- 移动端核心指标优先，长报表折叠或分组。
- 筛选、分页、导出、刷新和错误态统一。
- `includeSynthetic`、`statSource`、`perf-test` 口径必须明确。
- 长 referer、campaign、短码、hash 不得撑破布局。

## 组件规则

| 组件 | 规则 |
| --- | --- |
| Button | `min-height >= 44px`，主按钮深绿，次按钮白底边框，禁用态明确。 |
| Input / Select | `min-height >= 44px`，中文占位，移动端一列。 |
| Panel | 8px radius，轻边框，轻阴影，不做卡片套卡片。 |
| Chip | 可换行，不硬撑横向；颜色必须有文字语义。 |
| Table | 桌面宽表，移动端改卡片或折叠，不靠横向拖拽承载主信息。 |
| Disclosure | `<summary>` 也算触控目标，移动端至少 44px。 |
| Empty / Error | 文案说明下一步，不只显示空白或技术错误。 |

## 反目标

不要做：

- 大面积紫色或紫蓝渐变。
- 漂浮光斑、装饰球、虚焦背景。
- 营销 landing page 式大 hero 覆盖真实功能。
- 多层卡片嵌套。
- 过度圆角导致玩具感。
- 只靠颜色表达五行或状态。
- 页面能打开就宣称视觉完成。

## 验收

先选择模式：

- 开发改造默认 `FRONTEND_URL/E2E_BASE_URL=http://127.0.0.1:5175`，后端 `BACKEND_URL=http://127.0.0.1:48081`。
- 生产预览演示默认 `FRONTEND_URL/E2E_BASE_URL=http://127.0.0.1:4173`，后端仍为 `48081`，且后端 `APP_BASE_URL` 必须同步为 `4173`。

最低：

```bash
npm --prefix frontend run build
node scripts/verify-frontend-contracts.mjs
```

涉及主流程：

```bash
E2E_BASE_URL=http://127.0.0.1:5175 E2E_ADMIN_TOKEN=dev-token scripts/mobile-e2e.sh
```

涉及视觉变化：

```bash
E2E_BASE_URL=http://127.0.0.1:5175 E2E_ADMIN_TOKEN=dev-token scripts/capture-showcase-screenshots.sh
```

发布前：

```bash
REQUIRE_TRACKED_QUALITY_SCRIPTS=1 scripts/quality-check.sh
```

每次视觉交付都要记录截图路径、视口、DOM 复核、未验证项和残余风险。
