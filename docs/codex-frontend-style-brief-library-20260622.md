# Codex 前端 Style Brief 库

记录日期：2026-06-22
用途：把“我想要的前端样式”翻译成 Codex 可执行、可验收、可复用的设计约束。

## 使用方式

每次让 Codex 做视觉前端时，不要只写“好看一点”。至少给出一条 style brief：

```text
请使用 codex-frontend-workflow skill。
目标页面：
Style brief：
必须保留：
验收证据：
```

如果还没有明确审美，先让 Codex 从本库选 2-3 个方向，而不是直接改 CSS。

## Style Brief 结构

| 字段 | 要写什么 | 为什么重要 |
| --- | --- | --- |
| 场景 | 用户在哪个页面完成什么任务 | 决定信息密度和按钮优先级。 |
| 气质 | 3-5 个审美关键词 | 避免 Codex 随机发挥。 |
| 主色 | 主行动、背景、强调色 | 把颜色变成 token。 |
| 信息密度 | 稀疏、适中、高密度 | 区分 H5 和后台。 |
| 组件规则 | 按钮、卡片、表格、表单、弹窗 | 避免页面内部风格打架。 |
| 反目标 | 明确不要什么 | 比“高级”更容易执行。 |
| 验收 | 截图、DOM、E2E、无横向溢出 | 防止只凭感觉收口。 |

## 五行默认方向

```text
克制中式数据感：暖纸底、深绿主行动、五行色只做信息编码、移动端一屏一任务、后台高密度可扫描。
```

适用：

- 首页引导。
- 测试页。
- 结果页。
- 匹配页。
- 分享页。

执行规则：

- 背景用暖纸色或低对比纹理感，不做大面积纯白冷界面。
- 主行动用深绿，不用大面积紫蓝渐变。
- 五行色只用于元素标识、比例、标签和状态，不铺满背景。
- 结果页先让“人格身份”被看见，再解释五行比例。
- 文案情绪正向，但界面不做玄学压迫感。

反目标：

- 不做赛博算命。
- 不做土味古风。
- 不做卡片套卡片。
- 不用装饰性渐变球。
- 不把分享能力做成调试表单。

## H5 年轻化方向

```text
轻量社交 H5：清爽留白、强主行动、卡片式逐步完成、结果页有分享冲动，五行只作为人格标签。
```

适用：

- 想提升开始率、答完率、分享率时。
- 面向大学生、朋友传播、朋友圈截图场景。

执行规则：

- 首屏只保留一个主要行动。
- 每屏最多一个核心问题或核心结果。
- 按钮高度和触控目标优先保证移动端。
- 结果页标题比解释文更强，分享图入口明显但不吵。
- 空状态和错误态用短句，不写系统错误堆栈感文本。

反目标：

- 不做运营落地页式大 hero 文案。
- 不把所有结果解释一次性堆满。
- 不让按钮、标签、徽章都抢主行动。

验收重点：

- iPhone SE 视口不横向溢出。
- 主按钮不低于 44px 高。
- 结果页首屏能看见人格名称和分享入口。

## 后台数据中台方向

```text
安静高密度后台：浅底、深色正文、克制强调色、表格可扫描、筛选和导出优先，移动端不撑破。
```

适用：

- AdminDashboard。
- ShortLinkDetail。
- 统计、趋势、列表、导出。

执行规则：

- 页面不做营销英雄区。
- 统计卡只展示关键指标和变化趋势。
- 表格优先处理长 referer、campaign、shortUrl 和时间字段。
- 筛选区、表格区、详情区层级清楚，但不要嵌套多层卡片。
- 移动端允许列折叠、换行或详情化，不允许横向撑破视口。

反目标：

- 不使用过大的标题和装饰插图。
- 不用彩色背景块堆满页面。
- 不把导出、刷新、筛选做成同等视觉权重。

验收重点：

- 390px 宽无横向溢出。
- 长 URL / referer / campaign 不撑破。
- 筛选、导出、刷新按钮可点击且不会换行挤压。

## 截图还原方向

```text
参考截图优先：先提取布局、密度、色彩、字体层级和组件状态，再决定哪些地方按项目约束调整。
```

适用：

- 用户给截图、Figma、竞品页面。
- 目标是“接近这个感觉”，不是简单换颜色。

执行规则：

- 先描述截图的结构，不直接改。
- 标注必须完全对齐和可以项目化改写的部分。
- 将截图视觉转成 token、布局和组件规则。
- 真实实现后必须保存同视口截图对照。

反目标：

- 不照搬品牌 logo、版权图或商业素材。
- 不只换主色就声称匹配。
- 不忽略移动端重排。

验收重点：

- 桌面/移动截图都有。
- 关键间距、层级、状态和主行动能对上。
- 说明和参考图不同的地方为什么不同。

## 小修方向

```text
轻量视觉修复：只定位目标缺陷，保留现有视觉系统，最小 patch，做聚焦视口验证。
```

适用：

- 文本重叠。
- 横向溢出。
- 按钮太小。
- 表格撑破。
- 状态颜色不清楚。

执行规则：

- 不重新设计整页。
- 不引入新 UI 库。
- 不改无关 token。
- 验证目标 viewport 和具体元素。

验收重点：

- 缺陷在目标视口消失。
- 没有引入新的布局跳动。
- build 或对应轻量检查通过。

## 给 Codex 的可复制提示

```text
请使用 codex-frontend-workflow skill。

目标：优化 [页面/组件/链路]。
Style brief：选择 [五行默认方向 / H5 年轻化方向 / 后台数据中台方向 / 截图还原方向 / 小修方向]，并根据当前项目 token 落地。
必须保留：[路由/API 字段/testid/埋点/统计口径]
验收证据：[build/typecheck/E2E/screenshot/DOM overflow/QA 记录]

先输出 Task Frame 和需要读取的真实文件，然后再改代码。若 style brief 不够明确，先给 2-3 个方向让我选。
```
